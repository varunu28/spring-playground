# Spring Cloud Contract - Consumer-Driven Contracts

This project demonstrates Consumer-Driven Contract (CDC) testing with a **shared contracts module** in a monorepo structure.

## Project Structure

```
spring-cloud-contract/
├── contracts/                           # Shared contracts module (owned by consumers)
│   ├── pom.xml
│   └── src/main/resources/
│       └── contracts/
│           └── producer-service/        # ⚠️ IMPORTANT: Must match service name
│               └── fraudcheck-contract.groovy
│
├── producer-service/                    # Producer (implements the contracts)
│   ├── pom.xml
│   └── src/
│       ├── main/java/...
│       └── test/java/...
│
└── consumer-service/                    # Consumer (uses the stubs)
    ├── pom.xml
    └── src/
        ├── main/java/...
        └── test/
            ├── java/...
            └── resources/
                └── application.yml      # Stub runner configuration
```

**⚠️ CRITICAL**: Contracts MUST be in `contracts/<service-name>/` subdirectory, not directly in `contracts/`!

## How It Works

### 1. **Contracts Module** (Consumer Owned)
- **Location**: `contracts/src/main/resources/contracts/producer-service/`
- **Purpose**: Consumers define the contracts they expect from producers
- **Published**: As a Maven artifact (`com.varunu28:contracts:0.0.1-SNAPSHOT`)

### 2. **Producer Service** (Implements Contracts)
The producer:
1. Downloads contracts from the `contracts` module
2. Spring Cloud Contract auto-generates tests from contracts
3. Runs tests to verify producer implements the contract correctly
4. Publishes stubs if tests pass

**Key Configuration** (`producer-service/pom.xml`):
```xml
<dependency>
    <groupId>com.varunu28</groupId>
    <artifactId>contracts</artifactId>
    <version>0.0.1-SNAPSHOT</version>
    <scope>test</scope>
</dependency>

<plugin>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-contract-maven-plugin</artifactId>
    <configuration>
        <baseClassForTests>com.varunu28.producerservice.BaseTestClass</baseClassForTests>
        <testFramework>JUNIT5</testFramework>
        <contractsMode>REMOTE</contractsMode>
        <contractsRepositoryUrl>file://${user.home}/.m2/repository</contractsRepositoryUrl>
        <contractDependency>
            <groupId>com.varunu28</groupId>
            <artifactId>contracts</artifactId>
            <version>0.0.1-SNAPSHOT</version>
        </contractDependency>
        <contractsPath>contracts/producer-service</contractsPath>
    </configuration>
</plugin>
```

### 3. **Consumer Service** (Uses Stubs)
The consumer:
1. Runs tests using stubs generated by the producer
2. Stub runner automatically downloads and starts WireMock with producer stubs
3. Tests verify consumer works correctly with the contract

**Key Configuration** (`consumer-service/src/test/resources/application.yml`):
```yaml
stubrunner:
  ids: com.varunu28:producer-service:0.0.1-SNAPSHOT:stubs
  stubs-mode: remote
  repository-root: file://${user.home}/.m2/repository
```

## Workflow

### Initial Setup
```bash
# 1. Build and install contracts module
cd contracts
mvn clean install

# 2. Build producer (generates tests from contracts, creates stubs)
cd ../producer-service
mvn clean install

# 3. Run consumer tests (uses producer stubs)
cd ../consumer-service
mvn test
```

### When Consumer Wants to Change Contract

1. **Consumer modifies contract** in `contracts/src/main/resources/contracts/producer-service/`
2. **Consumer publishes new contract**:
   ```bash
   cd contracts
   mvn clean install
   ```
3. **Producer tries to build**:
   ```bash
   cd producer-service
   mvn clean test
   ```
   - ✅ If tests pass → Producer implements new contract correctly
   - ❌ If tests fail → Producer must update code to match contract

4. **Producer cannot publish stubs** until tests pass
5. **This prevents breaking changes!**

## Benefits of This Approach

### ✅ Consumer-Driven
- Consumers define what they need from producers
- Contracts live in a shared module, not buried in producer code

### ✅ Prevents Breaking Changes
- Producer **cannot** publish stubs if contract tests fail
- Contract changes require both consumer and producer agreement

### ✅ Monorepo Friendly
- Single repository for contracts
- No external services needed (Pact Broker, etc.)
- Works offline with local Maven repository

### ✅ Version Control
- Contracts are version-controlled
- Clear history of contract changes
- Easy to review contract modifications via PRs

## CI/CD Integration

Recommended pipeline:

```yaml
# Example CI/CD workflow
1. Build contracts module → Install to local repo
2. Build producer:
   - Downloads latest contracts
   - Runs contract tests
   - Fails build if tests fail
   - Publishes stubs only on success
3. Build consumer:
   - Downloads latest stubs
   - Runs integration tests
   - Fails if stubs don't match expectations
```

## Example Contract

```groovy
// contracts/src/main/resources/contracts/producer-service/fraudcheck-contract.groovy
package contracts

import org.springframework.cloud.contract.spec.Contract

Contract.make {
    request {
        method 'PUT'
        url '/fraudcheck'
        body([
            "client.id": $(regex('[0-9]{10}')),
            loanAmount: 99999
        ])
        headers {
            contentType('application/json')
        }
    }
    response {
        status OK()
        body([
            fraudCheckStatus: "FRAUD",
            "rejection.reason": "Amount too high"
        ])
        headers {
            contentType('application/json')
        }
    }
}
```

## What Prevents Breaking Changes?

The **contracts module** acts as the source of truth:

1. **Consumer owns the contract** → Defines expectations
2. **Producer must implement it** → Tests auto-generated from contract
3. **Producer tests fail** if implementation doesn't match contract
4. **No stubs published** if tests fail
5. **Consumer tests fail** if they try to use non-existent stubs

This creates a **contract-first** workflow where breaking changes are caught before deployment!
